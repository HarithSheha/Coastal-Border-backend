3 Database Setup

Create the MySQL database before running migrations. Open your MySQL client (MySQL Workbench, phpMyAdmin, or the terminal) and run:

CREATE DATABASE coastal_border CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
SHOW DATABASES; -- verify it appears in the list
⚠️
Make sure your MySQL server is running before creating the database. In XAMPP, start the MySQL service from the control panel.

Open .env in the project root and fill in your MySQL credentials:

ℹ️
If your MySQL root user has no password (default XAMPP setup), leave DB_PASSWORD empty.

5 Run Migrations & Seed Data

Migrations create all the database tables. The seeder inserts sample data (5 zones, 8 sensors, 8 live reports, 80 sensor readings). The new reports and urgencies tables are not seeded — create urgency levels and sighting reports yourself via the API (see §8).

# Run migrations only (creates tables)
php artisan migrate

# Run migrations AND seed sample data (recommended for first run)
php artisan migrate --seed

# Reset everything and start fresh
php artisan migrate:fresh --seed

After a successful run you will see:

INFO  Running migrations.
  2024_01_01_000010_create_zones_table .............................. 25ms DONE
  2024_01_01_000011_create_sensors_table ............................ 31ms DONE

⚠️
The original 2024_01_01_000012_create_reports_table migration still creates a table literally named reports — the very next migration (rename_reports_table_to_live_reports) immediately renames it to live_reports, freeing up the reports name for the new sighting-reports table created afterwards. Migrations must run in this order; don't reorder or skip them.

7 API Endpoints Reference

Zones

MethodURLDescriptionBody Required
GET/api/zonesList all zones (includes sensor & report counts)—
POST/api/zonesCreate a new zonename, type
GET/api/zones/{id}Get zone with its sensors and reports—
PUT/api/zones/{id}Update zone fieldsany field
DELETE/api/zones/{id}Delete zone (sensors/reports become zone-less)—

Sensors

MethodURLDescriptionFilters / Body
GET/api/sensorsList all sensors with zone?status=alert ?zone_id=
POST/api/sensorsRegister a new sensorname, type
GET/api/sensors/{id}Get sensor with zone + last 50 readings—
PUT/api/sensors/{id}Update sensor (e.g. change status)any field
DELETE/api/sensors/{id}Remove sensor—

Reports sighting reports

Public/field sighting reports — a person seen, a boat spotted, etc. — filed against a zone with an urgency level. Primary key is report_id (auto-increment integer), not a UUID.

MethodURLDescriptionFilters / Body
GET/api/reportsList all reports with zone & urgency?zone_id= ?urgency_id= ?date= ?search=
POST/api/reportsSubmit a new sighting reportdate, latitude, longitude, address, zone_id, color, number_of_people, name, phone, urgency_id
GET/api/reports/{report_id}Get single report with relations—
PUT/api/reports/{report_id}Update report fieldsany field
DELETE/api/reports/{report_id}Delete a report—

Urgencies

Lookup table referenced by reports.urgency_id (e.g. "Low", "Medium", "High", "Critical" — seed whatever levels you need).

MethodURLDescriptionFilters / Body
GET/api/urgenciesList all urgency levels—
POST/api/urgenciesCreate a new urgency levelurgency_level
GET/api/urgencies/{urgency_id}Get a single urgency level—
PUT/api/urgencies/{urgency_id}Rename an urgency levelurgency_level
DELETE/api/urgencies/{urgency_id}Delete an urgency level—

Live Reports incident / sensor alerts

This is what used to be called Reports before the sighting-reports table above took over the /api/reports name. Same UUID-based schema as before, just moved to /api/live-reports.

MethodURLDescriptionFilters / Body
GET/api/live-reportsList all live reports with zone & sensor?status=open ?severity=critical ?type=intrusion ?search=
POST/api/live-reportsSubmit a new incident reporttitle, type, severity, source, reporter_name
GET/api/live-reports/{id}Get single live report with relations—
PUT/api/live-reports/{id}Update live report (e.g. change status)any field
DELETE/api/live-reports/{id}Delete a live report—

Sensor Readings

MethodURLDescriptionFilters / Body
GET/api/sensor-readingsList latest 100 readings?sensor_id= ?triggered=1
POST/api/sensor-readingsLog a new sensor readingsensor_id, value, unit
GET/api/sensor-readings/{id}Get a single reading—

Option B — cURL (all request types)

GET all zones:

curl -X GET http://localhost:8000/api/zones \
     -H "Accept: application/json"

POST — Create a new zone:

curl -X POST http://localhost:8000/api/zones \
     -H "Content-Type: application/json" \
     -H "Accept: application/json" \
     -d '{
       "name": "Test Zone Alpha",
       "description": "Testing zone creation",
       "type": "caution",
       "status": "active",
       "color": "#f59e0b"
     }'

PUT — Update a zone status to breach:

curl -X PUT http://localhost:8000/api/zones/{zone-id-here} \
     -H "Content-Type: application/json" \
     -H "Accept: application/json" \
     -d '{"status": "breach"}'

POST — Create an urgency level (do this first — reports need one):

curl -X POST http://localhost:8000/api/urgencies \
     -H "Content-Type: application/json" \
     -H "Accept: application/json" \
     -d '{"urgency_level": "High"}'

POST — Submit a sighting report:

curl -X POST http://localhost:8000/api/reports \

PUT — Update a report's urgency:

curl -X PUT http://localhost:8000/api/reports/{report_id-here} \
     -H "Content-Type: application/json" \

curl -X POST http://localhost:8000/api/live-reports \

Option C — Postman (Recommended)

  1. Download and install Postman from postman.com
  2. Create a new Collection called Coastal Border API
  3. Add a Collection Variable: base_url = http://localhost:8000/api
  4. For every request, set the header: Accept: application/json
  5. For POST/PUT requests, set: Content-Type: application/json and use the Body → raw → JSON tab
  6. Send GET {{base_url}}/zones — you should see the 5 seeded zones
  7. Copy a zone id from the response and test GET {{base_url}}/zones/{id}
  8. Test filtering: GET {{base_url}}/live-reports?severity=critical
✅
A successful response returns HTTP 200 OK (GET/PUT), 201 Created (POST), or 204 No Content (DELETE). Validation errors return 422 Unprocessable Entity with a JSON errors object.

9 Connecting to the React Frontend

The React frontend currently fetches data directly from Supabase. To switch it to use the Laravel backend instead, follow these steps.

Step 1 — Create an API client file in the frontend

In the React project at src/lib/api.ts, create this file:

// src/lib/api.ts
const BASE_URL = 'http://localhost:8000/api';

async function request(path: string, options: RequestInit = {}) {
  const res = await fetch(`${BASE_URL}${path}`, {
    headers: { 'Content-Type': 'application/json', 'Accept': 'application/json' },
    ...options,
  });
  if (!res.ok) throw new Error(`API error ${res.status}`);
  if (res.status === 204) return null;
  return res.json();
}

export const api = {
  zones:   { list: () => request('/zones'),            create: (d: any) => request('/zones', { method: 'POST', body: JSON.stringify(d) }),
             update: (id: string, d: any) => request(`/zones/${id}`, { method: 'PUT', body: JSON.stringify(d) }),
             remove: (id: string) => request(`/zones/${id}`, { method: 'DELETE' }) },

  sensors: { list: (params?: string) => request(`/sensors${params || ''}`),
             create: (d: any) => request('/sensors', { method: 'POST', body: JSON.stringify(d) }),
             update: (id: string, d: any) => request(`/sensors/${id}`, { method: 'PUT', body: JSON.stringify(d) }),
             remove: (id: string) => request(`/sensors/${id}`, { method: 'DELETE' }) },

  // New sighting reports (report_id, zone_id, urgency_id, ...)
  reports: { list: (params?: string) => request(`/reports${params || ''}`),
             create: (d: any) => request('/reports', { method: 'POST', body: JSON.stringify(d) }),
             update: (id: number, d: any) => request(`/reports/${id}`, { method: 'PUT', body: JSON.stringify(d) }),
             remove: (id: number) => request(`/reports/${id}`, { method: 'DELETE' }) },

  urgencies: { list: () => request('/urgencies'),
             create: (d: any) => request('/urgencies', { method: 'POST', body: JSON.stringify(d) }) },

  // Old incident/sensor-alert reports, moved from /reports to /live-reports
  liveReports: { list: (params?: string) => request(`/live-reports${params || ''}`),
             create: (d: any) => request('/live-reports', { method: 'POST', body: JSON.stringify(d) }),
             update: (id: string, d: any) => request(`/live-reports/${id}`, { method: 'PUT', body: JSON.stringify(d) }),
             remove: (id: string) => request(`/live-reports/${id}`, { method: 'DELETE' }) },

  readings:{ list: (params?: string) => request(`/sensor-readings${params || ''}`),
             create: (d: any) => request('/sensor-readings', { method: 'POST', body: JSON.stringify(d) }) },
};

Step 2 — Replace Supabase calls in App.tsx

Open src/App.tsx and replace the fetchData function:

// Remove this import:
// import { supabase } from './lib/supabase';

// Add this import:
import { api } from './lib/api';

// Replace fetchData with:
const fetchData = useCallback(async (isRefresh = false) => {
  if (isRefresh) setRefreshing(true);
  // NOTE: the dashboard's incident-alert feed is now api.liveReports (was api.reports)
  // api.reports is now the new zone/urgency sighting-reports endpoint — wire it up separately.
  const [reportData, sensorData, zoneData] = await Promise.all([
    api.liveReports.list(),
    api.sensors.list(),
    api.zones.list(),
  ]);
  if (reportData) setReports(reportData);
  if (sensorData) setSensors(sensorData);
  if (zoneData)   setZones(zoneData);
  setLoading(false);
  if (isRefresh) setRefreshing(false);
}, []);

Step 3 — Replace Supabase calls in each page

Each page that writes to Supabase also needs to be updated. Example for Reports.tsx (the incident-alert page — this maps to live-reports, not the new reports table):

// Old (Supabase):
await supabase.from('reports').update({ status }).eq('id', id);

// New (Laravel API):
import { api } from '../lib/api';
await api.liveReports.update(id, { status });

Example for Sensors.tsx — adding a sensor:

// Old (Supabase):
await supabase.from('sensors').insert({ name, type, zone_id, status: 'online', battery_level: 100, ... });

// New (Laravel API):
await api.sensors.create({ name, type, zone_id, status: 'online', battery_level: 100 });

Step 4 — Add the BASE_URL to the frontend .env

In the React project root, add to .env:

VITE_API_URL=http://localhost:8000/api

Then update src/lib/api.ts to use it:

const BASE_URL = import.meta.env.VITE_API_URL || 'http://localhost:8000/api';
⚠️
The backend's CORS config already allows http://localhost:5173 (Vite's default port). If your frontend runs on a different port, add it to config/cors.php in the backend under allowed_origins.

Step 5 — Start both servers

# Terminal 1 — Backend
cd "...Coastal-Border-backend"
php artisan serve
# → http://localhost:8000

# Terminal 2 — Frontend
cd "...Coastal-border-"
npm run dev
# → http://localhost:5173
✅
With both servers running, the React dashboard will fetch all its data from the Laravel MySQL backend instead of Supabase.

10 Troubleshooting

❌ SQLSTATE[HY000] [2002] No connection

MySQL is not running. Start it from XAMPP Control Panel or run net start MySQL80 in an admin terminal.

❌ Access denied for user 'root'@'localhost'

Wrong MySQL password in .env. Check your MySQL credentials and update DB_PASSWORD.

❌ 404 on /api/zones

The server is not running. Make sure php artisan serve is still active in your terminal. Do not close that terminal window.

❌ CORS error in the browser console

Open config/cors.php in the backend and add your frontend's URL to allowed_origins, then restart the server.

❌ 422 Unprocessable Entity on POST

A required field is missing or invalid. The response body contains a JSON errors object listing exactly which fields failed validation.

❌ php artisan migrate fails

Run php artisan migrate:status to see which migrations failed. Most likely the database does not exist yet — create it first with CREATE DATABASE coastal_border;
